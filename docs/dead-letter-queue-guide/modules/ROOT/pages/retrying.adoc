= Retrying Failed Dead-Letter Sequences
:navtitle: Retrying failed sequences

The dead letter queue is a stop-gap. It's important to strive to have the queue empty as soon as possible. Ideally by retrying the events, and removing them from the queue.

You can retrieve `SequencedDeadLetterProcessor` instances from the `Configuration` based on a processor name.
Each Event Handling Component with DLQ enabled is registered as a `SequencedDeadLetterProcessor`.

[TIP]
You can also use AxonIQ Platform to inspect and replay dead-lettered events through the UI without writing code.

== Scheduling retries

A common pattern is to schedule periodic retries with a fixed delay, giving the system time to recover between attempts:

[source,java]
----
import org.axonframework.common.configuration.Configuration;
import org.axonframework.messaging.deadletter.SequencedDeadLetterProcessor;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Component;

@Component
public class DeadLetterRetryScheduler {

    private final Configuration configuration;

    public DeadLetterRetryScheduler(Configuration configuration) {
        this.configuration = configuration;
    }

    @Scheduled(fixedDelay = 30_000, initialDelay = 30_000) // <1>
    public void retryDeadLetters() {
        configuration.findModules(SequencedDeadLetterProcessor.class) // <2>
                .forEach(processor ->
                        processor.processAny() // <3>
                                .orTimeout(30, java.util.concurrent.TimeUnit.SECONDS)
                                .join());
    }
}
----

<1> Spring-specific way to wait 30 seconds after startup, then invoke the method every 30 seconds.
Use a generous interval to avoid stressing a system that may already be under pressure.
<2> Retrieves all registered `SequencedDeadLetterProcessor` instances across all processors and components.
You can also filter by processor name if you want different retry intervals for different processors.
<3> `processAny()` picks the oldest unprocessed sequence.
Returns `true` if a sequence was processed successfully, `false` if there were no sequences to process or processing failed.
Once there are multiple failing sequences, `processAny()` rotates through them on each call.

== Processing a specific sequence

To retry only sequences matching a particular event type, use `process(Predicate)`:

[source,java]
----
import org.axonframework.common.configuration.Configuration;
import org.axonframework.messaging.core.QualifiedName;
import org.axonframework.messaging.deadletter.DeadLetter;
import org.axonframework.messaging.deadletter.SequencedDeadLetterProcessor;
import org.axonframework.messaging.eventhandling.EventMessage;
import java.util.concurrent.TimeUnit;

public class DeadLetterRetryScheduler {

    // ErrorEvent is your application-defined event class
    private static final QualifiedName ERROR_EVENT_TYPE = new QualifiedName(ErrorEvent.class);

    private final Configuration configuration;

    public DeadLetterRetrier(Configuration configuration) {
        this.configuration = configuration;
    }

    public void retryErrorEventSequences(String processorName) {
        configuration.getModuleConfiguration(processorName)
                .map(m -> m.getComponents(SequencedDeadLetterProcessor.class))
                .ifPresent(processors -> processors.values().forEach(dlp ->
                        dlp.process(letter -> letter.message()
                                                    .getMessageType()
                                                    .equals(ERROR_EVENT_TYPE))
                           .orTimeout(30, TimeUnit.SECONDS)
                           .join()));
    }
}
----

== Processing all sequences until empty

To drain the entire queue, for example after deploying a fix, process sequences in a loop until none remain:

[source,java]
----
import org.axonframework.common.configuration.Configuration;
import org.axonframework.messaging.deadletter.SequencedDeadLetterProcessor;
import java.util.concurrent.TimeUnit;

public class DeadLetterRetryScheduler {

    private final Configuration configuration;

    public DeadLetterDrainer(Configuration configuration) {
        this.configuration = configuration;
    }

    public void drainAll(String processorName) {
        configuration.getModuleConfiguration(processorName)
                .map(m -> m.getComponents(SequencedDeadLetterProcessor.class))
                .ifPresent(processors -> processors.values()
                        .forEach(this::processUntilEmpty));
    }

    private void processUntilEmpty(SequencedDeadLetterProcessor processor) {
        boolean processed;
        do {
            processed = processor.processAny()
                    .orTimeout(30, TimeUnit.SECONDS)
                    .join();
        } while (processed);
    }
}
----

By default, the framework re-enqueues a letter if processing fails again.

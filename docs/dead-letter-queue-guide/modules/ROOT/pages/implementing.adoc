= Implementing Dead Letter Queues (DLQ)
:navtitle: Implementing dead letter queue

To enable a DLQ you configure the dead-letter queue on the event processor.
Axon Framework includes JPA and JDBC implementations of `SequencedDeadLetterQueue`.
To best leverage transactions, it's best to choose the same implementation as the `TokenStore` and likely also for your projection. Please note that for best results, this also means having them located in the same database.

[NOTE]
====
Before enabling a DLQ, verify that your event handlers are idempotent.
When a failed event is retried, all event handlers within the Event Handling Component that originally failed are invoked again.
If handlers are not idempotent, retrying may produce undesired side effects.
====

== Enabling the dead-letter queue

[tabs]
====
Configuration API::
+
--
Use `EventProcessorModule.customized()` to enable the DLQ for a specific processor.
The example below uses `JpaSequencedDeadLetterQueue`:

[source,java]
----
import org.axonframework.messaging.core.configuration.MessagingConfigurer;
import org.axonframework.messaging.eventhandling.configuration.EventProcessorModule;
import org.axonframework.messaging.eventhandling.deadletter.jpa.JpaSequencedDeadLetterQueue;
import org.axonframework.messaging.core.unitofwork.transaction.jpa.JpaTransactionalExecutorProvider;
import org.axonframework.messaging.eventhandling.conversion.EventConverter;
import org.axonframework.conversion.Converter;
import jakarta.persistence.EntityManagerFactory;

public class AxonConfig {

    public void configureDeadLetterQueue(
            EntityManagerFactory entityManagerFactory,
            EventConverter eventConverter,
            Converter converter
    ) {
        var configurer = MessagingConfigurer.create();
        configurer.eventProcessing(ep -> ep.pooledStreaming(ps -> ps
                .processor(
                        EventProcessorModule
                                .pooledStreaming("my-processor")
                                .eventHandlingComponents(components -> components
                                        .declarative("myHandler", cfg -> myHandlerComponent))
                                .customized((cfg, processorConfig) -> processorConfig
                                        .deadLetterQueue(dlq -> dlq
                                                .enabled()
                                                .factory((name, config) ->
                                                        JpaSequencedDeadLetterQueue.builder()
                                                                .processingGroup(name)
                                                                .maxSequences(256)
                                                                .maxSequenceSize(256)
                                                                .transactionalExecutorProvider(
                                                                        new JpaTransactionalExecutorProvider(entityManagerFactory))
                                                                .eventConverter(eventConverter)
                                                                .genericConverter(converter)
                                                                .build())))
                )
        ));
    }
}
----
--

Spring Boot::
+
--
The simplest way to enable the DLQ in Spring Boot is via properties.
Spring Boot auto-configuration automatically detects whether JPA or JDBC is available and creates the appropriate `SequencedDeadLetterQueueFactory` bean:

[source,properties]
----
axon.eventhandling.processors.my-processor.dlq.enabled=true
----

To customize queue capacity or use a non-default implementation, define a `SequencedDeadLetterQueueFactory` bean.
The factory is called once per Event Handling Component and receives the queue name as its first argument (following the `DeadLetterQueue[processorName][beanName]` pattern, where `beanName` is the Spring bean name of the event handler):

[source,java]
----
import org.axonframework.conversion.Converter;
import org.axonframework.messaging.eventhandling.conversion.EventConverter;
import org.axonframework.messaging.eventhandling.deadletter.SequencedDeadLetterQueueFactory;
import org.axonframework.messaging.eventhandling.deadletter.jpa.JpaSequencedDeadLetterQueue;
import org.axonframework.messaging.core.unitofwork.transaction.jpa.JpaTransactionalExecutorProvider;
import jakarta.persistence.EntityManagerFactory;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class DlqConfig {

    @Bean
    SequencedDeadLetterQueueFactory myDeadLetterQueueFactory(
            EntityManagerFactory entityManagerFactory,
            EventConverter eventConverter,
            Converter converter
    ) {
        return (processingGroup, config) ->
                JpaSequencedDeadLetterQueue.builder()
                        .processingGroup(processingGroup)
                        .maxSequences(256)
                        .maxSequenceSize(256)
                        .transactionalExecutorProvider(
                                new JpaTransactionalExecutorProvider(entityManagerFactory))
                        .eventConverter(eventConverter)
                        .genericConverter(converter)
                        .build();
    }
}
----

[source,properties]
----
axon.eventhandling.processors.my-processor.dlq.enabled=true
----
--

== Queue naming

Each Event Handling Component within a processor gets its own queue instance.
Queues are identified by the pattern `DeadLetterQueue[processorName][componentName]`, where `componentName` is the name passed to `declarative(...)` / `autodetected(...)` in the Configuration API, or the Spring bean name in Spring Boot.

Keep component names stable.
Changing a component name orphans any previously dead-lettered events.

== Capacity limits

You can set the maximum number of sequences (defaults to 1024) and the maximum number of letters per sequence (also defaults to 1024).
If either threshold is exceeded, the queue throws a `DeadLetterQueueOverflowException` and the processor stops accepting new events, falling back to the xref:axon-framework-reference:events:event-processors/index.adoc#error-handling[default error handling behavior].

Although this enables the processor to continue in case of errors, it does not retry failed events automatically.
The next section explains how to enable retries.
